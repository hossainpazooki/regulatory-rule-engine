# Gate-5 visual-regression harness (Workstream E)

Self-hosted Playwright full-page screenshot suite for the genuinely-rewired
(flag-on) surfaces and the 5e AI-provenance review UI. No live backend: every
network call is mocked deterministically.

> **Status: experimental / non-gating.** This harness is not a Gate-5 acceptance
> gate. The real 5d/5e gate is `npm run typecheck` + `test:run` + `build` green
> with flags default-off (`main` unchanged). Treat pixel diffs as advisory.

## What it covers

| Spec name | Route | Why |
|---|---|---|
| `workbench.png` | `/workbench` | Local KE preview pane rewired (compile/dry-run/verify, serve); rule LIST stays on `VITE_API_URL` fallback (scaffold-only) |
| `production.png` | `/production` | Health rewired (serve `GET /healthz`) |
| `review-ui-workbench.png` | `/workbench` (`USE_REVIEW_UI=true`) | 5e review UI in the preview pane, after a verify |

The 7 ML/analytics off-path pages (analytics, embeddings, graph, navigate,
documents, similarity, home) are intentionally **not** snapshotted: per **ADR-0020**
they stay on the external `VITE_API_URL` by design (off the ATLAS artifact path,
ADR-0017), so flag-on they render identically to today — a baseline carries no
signal.

## Determinism

- **No live backend.** `visual.spec.ts` installs `page.route()` mocks from
  `fixtures/handlers.ts` (`routeTable`) for both mocked origins —
  `VITE_API_URL` (`http://msw.local/api`) and `KE_SERVE_URL`
  (`http://localhost:8787`). The same bodies are also exported as MSW v2
  `handlers` for the service-worker path (`public/mockServiceWorker.js`,
  generated by `npx msw init`).
- **WASM never loads in CI.** The build runs with
  `VITE_USE_WASM_PREVIEW=false`, so the rewired path is exercised over
  MSW-mocked serve HTTP and the `.wasm` binary is never fetched (convention B).
- **Pinned render surface:** viewport `1280x800`, `deviceScaleFactor: 1`,
  `colorScheme: 'light'`, animations disabled (see `playwright.config.ts`).

## CRITICAL: baselines are canonical on LINUX only

Cross-OS font hinting and anti-aliasing differ enough that a Windows-generated
PNG will not match a Linux baseline pixel-for-pixel. Therefore:

- **The `ubuntu-latest` CI leg (`.github/workflows/frontend-visual.yml`) is the
  single pixel authority.** Committed baselines under `e2e/__screenshots__/` are
  authoritative only if they were generated on Linux.
- **Do NOT commit Windows-generated baselines as canonical.** A Windows-local
  `npm run e2e` is advisory only — useful to confirm the suite runs and to eyeball
  layout, but its pixels are not the reference.
- The single Playwright project is named `linux`; `snapshotPathTemplate` bakes
  `-linux` into every committed baseline filename, so a Windows run reads/writes
  a differently-named file and can never silently overwrite the canonical PNGs.

### Baseline bootstrap (first run, on Linux)

There are no committed baselines yet. Generate them on the Linux CI leg so they
are pixel-authoritative from the start. Two equivalent ways:

1. **Via CI (preferred).** On the first CI run there are no baselines, so the
   `Visual regression` step will fail with "missing snapshot". Run the suite once
   in update mode on a Linux runner — locally in the Playwright Linux container,
   or by temporarily running `e2e:update` on `ubuntu-latest` — then commit the
   produced `e2e/__screenshots__/**.png`:

   ```bash
   # in the official Playwright Linux image, from frontend/
   VITE_USE_LOCAL_KE_API=true \
   VITE_USE_REVIEW_UI=true \
   VITE_USE_WASM_PREVIEW=false \
   VITE_API_URL=http://msw.local/api \
     npm run build
   npm run e2e:update
   git add e2e/__screenshots__
   ```

   ```bash
   # equivalently, pull and run the pinned Playwright Linux image directly
   docker run --rm -v "$PWD":/work -w /work/frontend \
     mcr.microsoft.com/playwright:v1.61.0-jammy \
     bash -c 'npm ci && VITE_USE_LOCAL_KE_API=true VITE_USE_REVIEW_UI=true \
       VITE_USE_WASM_PREVIEW=false VITE_API_URL=http://msw.local/api npm run build \
       && npm run e2e:update'
   ```

2. After the baselines are committed, every later CI run **verifies** against
   them. CI never auto-updates or auto-commits baselines; refreshing them after an
   intentional UI change is always an explicit `npm run e2e:update` on Linux,
   reviewed and committed by a human.

## Local (Windows) usage — advisory only

```bash
# from frontend/ — build with the flags baked, then run the suite
VITE_USE_LOCAL_KE_API=true VITE_USE_REVIEW_UI=true VITE_USE_WASM_PREVIEW=false \
  VITE_API_URL=http://msw.local/api npm run build
npm run e2e            # compares against linux baselines — pixel diffs EXPECTED on Windows
npm run e2e -- --list  # just list the specs (no browser needed)
```

A Windows diff failure does **not** mean the UI regressed — it usually means
cross-OS rendering drift. Trust the Linux CI leg.

## Scripts

- `npm run e2e` — run the suite (verify against baselines).
- `npm run e2e:update` — write/refresh baselines. **Run on Linux only.**
- `npm run e2e:ci` — CI reporter (`--reporter=github`).
