# 0022. Kind-aware R7 co-attestation — an IntentSpec approval co-attests with SourceFidelity only

**Status:** Proposed (draft for review)
**Date:** 2026-07-12
**Spec references:** § 10 (attestation rules), `docs/attestation-schema.md` § 6B/§ 7 (R7)
**Amends:** the R7 co-attestation rule in `verify_attestation_set` — from a fixed `{ScenarioCoverage, SourceFidelity}` pair to a kind-selected set.
**Related ADRs:** 0021 § 5 (IntentSpec attestation set = `SourceFidelity` + `PublicationApproval`), 0011 (T2/T3 policy).

## Context

R7 (schema § 6B) honors a valid `publication_approval` only when co-attestations
over the same artifact hash exist. As implemented it demanded a **fixed** pair —
`scenario_coverage` **and** `source_fidelity` — for every artifact kind.

Accepted ADR-0021 § 5 pins the `IntentSpec` attestation set to **exactly**
`SourceFidelity` + `PublicationApproval`: `ScenarioCoverage` is a rule-scenario
concept with no meaning for authorization criteria, and `ke-cli`'s kind-aware
publish policy (`intentspec_verification_policy`, crates/ke-cli/src/policy.rs)
already omits it from R6's required set.

These two rules are contradictory in composition: R6's kind-aware policy admits
the two-type set, then the unconditional R7 rejects it —
`CoAttestationAbsent { missing: ScenarioCoverage }`. **No IntentSpec artifact
can pass `verify_attestation_set`, so none can publish or verify, by
construction.** The gap was latent because no test drove an IntentSpec through
the attestation gate: the golden (`fixtures/artifacts/intentspec_payment`) is
generated by `gen-golden-artifacts` (which appends attestations without
verifying the set), the 3-language contract test compares the three legs'
verdicts for *equality* (all three said `rejected:` identically), and no
ke-cli test publishes an IntentSpec. It surfaced when the treasury resolver
(the ADR-0021 single consumer) first ran the folded verify against the golden
under the kind-correct policy — 2026-07-12.

## Decision

Make R7's co-attestation set kind-selected, exactly parallel to the R6 policy
split `policy_for_kind` already applies:

- **Rule-shaped kinds** (every non-`IntentSpec` kind): `scenario_coverage` +
  `source_fidelity` — unchanged, schema § 6B.
- **`IntentSpec`**: `source_fidelity` only (ADR-0021 § 5).

Implemented as a private `co_attestation_types(&ArtifactKind)` selector in
`crates/ke-artifact/src/attestation.rs`; `verify_attestation_set` consults the
artifact's `manifest.artifact_kind`, which the decode path already guarantees
agrees with the payload (ADR-0021 § Decision-4 kind↔payload agreement).

## Consequences

- The ADR-0021 two-type attestation set becomes honorable: an attested
  IntentSpec verifies and publishes through the same gate as every other kind.
- All three contract-test legs (Rust / Python wheel / WASM) shift together —
  they share this Rust core — so the cross-language equality gate is
  unaffected.
- Existing rule-kind behavior is byte-identical; the R7 rejection matrix tests
  for rule artifacts pass unchanged. Two new tests pin the IntentSpec arm
  (accepted with SourceFidelity; `CoAttestationAbsent { SourceFidelity }`
  without it, and ScenarioCoverage never demanded).
- The schema doc (`docs/attestation-schema.md` § 6B) still states the fixed
  pair; amend it to reference the kind-selected set when this ADR is accepted.
